# Synthetic Robustness And Split-Aware Validation Plan

Date: 2026-06-12

This plan defines the documentation gate for future synthetic or committed
local-fixture robustness summaries. It does not implement code, modify tests,
regenerate reports, fetch data, change strategy behavior, change backtester
behavior, or interpret results.

The plan applies only to simulated research artifacts. It does not authorize
real data, vendor APIs, credentials, live trading, paper trading, brokerage
integration, order execution, investment advice, or profitability claims.

## 1. Problem

The repository now has split helpers, synthetic diagnostics, local-fixture
workflows, benchmark-aware simulated backtests, fixed-bps transaction costs,
fixed-bps slippage, and a diagnostic/precomputed volume-aware slippage
boundary. The remaining process gap is robustness discipline:

- future summaries need to report all configured cases, not only favorable
  cases.
- train, validation, and test periods need explicit chronological boundaries.
- benchmark, universe, cost, slippage, and execution-timing assumptions need
  to remain inspectable.
- synthetic/local-fixture outputs need caveats that prevent real-data,
  execution-readiness, or profitability framing.

Without this plan, a future implementation could accidentally present a
parameter sweep, split diagnostic, or local-fixture smoke check as stronger
evidence than it is.

## 2. Scope

In scope for a future implementation:

- synthetic panels generated by repository research scripts.
- committed local fixtures used for smoke tests and workflow demonstrations.
- deterministic parameter grids that are declared before running a summary.
- chronological train, validation, and test windows.
- all-case tables that include every configured parameter case.
- caveated report and experiment-log fields for synthetic/local-fixture
  diagnostics.

Out of scope:

- real data fetching or downloads.
- user-provided local CSV interpretation without the readiness audit.
- vendor APIs, including market data vendors or brokerage APIs.
- credentials, tokens, `.env`, account IDs, or secret handling.
- live trading, paper trading, brokerage integration, order execution, or
  order-routing simulation.
- claims that any factor is profitable, trade-ready, production-ready, or a
  verified stock-selection signal.
- bulk WorldQuant 101 implementation.

## 3. Required Inputs

A future robustness summary should require explicit inputs before it runs:

- data scope: `synthetic` or `committed_local_fixture`.
- source artifact names and deterministic seeds, when synthetic.
- feature names, factor-combination method, and parameter grid.
- target return definition and signal lag.
- chronological split windows with start date, end date, and label.
- benchmark definition and benchmark availability window.
- universe or liquidity mask definition, if used.
- transaction-cost basis points and fixed-slippage basis points.
- volume-aware slippage mode: diagnostic-only, precomputed-impact, or absent.
- rebalance frequency and execution timing assumption.
- minimum observation rules for each split and metric.
- missing-data policy and incomplete-window policy.
- report ID, experiment ID, and run timestamp source.

All inputs must be deterministic and reviewable from committed code, committed
fixtures, or an explicit future run configuration. No hidden manual edits to
reported metrics are allowed.

## 4. Split Policy

Future split-aware summaries must use chronological, non-overlapping windows.
The implementation must preserve the project rule that signal inputs are known
before execution or target-return measurement.

Required split behavior:

- every row used in a metric must be assigned to exactly one split.
- train, validation, and test windows must be reported even when a split has
  insufficient data.
- test results must not be used to choose parameters.
- validation results may be used only for documented diagnostic comparison,
  not as proof of investment performance.
- same-period target returns must not be used as features.
- feature availability, signal date, execution date, and target-return window
  must remain distinct.

Stop if date alignment cannot be proven with deterministic tests.

## 5. Robustness Reporting Policy

Future summaries must show the full configured evaluation surface. They must
not hide weak cases, failed cases, missing cases, or parameter settings that
underperform other settings.

Required report behavior:

- include every configured parameter case.
- include every configured split for every case.
- include missing/invalid case counts and reasons.
- sort by a stable non-performance key by default, such as parameter tuple or
  experiment ID.
- if any performance-like metric is sorted, label it diagnostic-only and still
  include the full unsorted case table.
- report benchmark assumptions alongside strategy diagnostics.
- report transaction cost and slippage assumptions separately.
- report volume-aware slippage diagnostics or precomputed impacts separately
  from fixed-bps transaction costs and fixed-bps slippage.
- state that synthetic/local-fixture results are not real-market evidence.

## 6. Missing Data And Stop Conditions

Future implementation should fail closed on ambiguity. It must not silently
fill or drop data in a way that changes the research interpretation.

Stop conditions:

- missing split boundaries.
- overlapping split windows.
- split windows outside the available panel.
- target returns not lagged relative to signal inputs.
- feature rows with unknown availability timing.
- missing benchmark data when benchmark-relative metrics are requested.
- missing or invalid cost/slippage assumptions.
- missing volume data when volume-aware diagnostics are requested.
- incomplete rolling windows unless the configuration explicitly reports them
  as insufficient rather than treating them as valid metrics.
- all candidate cases invalid or empty.
- any requirement for real data, external downloads, vendor APIs,
  credentials, or brokerage/order functionality.
- wording that frames synthetic/local-fixture output as profitability evidence.

Allowed behavior for insufficient data is to record the case as invalid or
insufficient with a clear reason and to continue only when the report remains
complete and non-misleading.

## 7. Required Future Tests

Before any implementation PR, tests should be planned for:

- non-overlapping chronological split assignment.
- feature-date, signal-date, execution-date, and target-return alignment.
- rejection of overlapping, missing, unsorted, or empty split windows.
- reporting every configured parameter case.
- reporting every configured split for each case.
- no-best-only behavior, including cases with weak or negative diagnostics.
- insufficient observation handling.
- missing benchmark handling.
- fixed-bps transaction cost and fixed-bps slippage fields remaining separate.
- volume-aware diagnostic/precomputed-impact fields remaining separate from
  fixed-bps cost and fixed-bps slippage.
- deterministic report ordering.
- caveat text for synthetic/local-fixture diagnostics.
- guardrails blocking real data, vendor APIs, credentials, live/paper trading,
  brokerage, order execution, and profitability claims.

## 8. Required Future Experiment-Log Fields

Future experiment-log payloads should include:

- `data_scope`.
- `source_artifacts`.
- `synthetic_seed`, when applicable.
- `split_policy`.
- `split_windows`.
- `parameter_grid`.
- `reported_case_count`.
- `invalid_case_count`.
- `invalid_case_reasons`.
- `benchmark_assumption`.
- `rebalance_frequency`.
- `execution_timing`.
- `target_return_definition`.
- `signal_lag`.
- `transaction_cost_bps`.
- `slippage_bps`.
- `volume_aware_slippage_mode`.
- `metrics_by_split`.
- `cost_fields`.
- `slippage_fields`.
- `guardrail_caveats`.

Field names may be refined during implementation, but the payload must keep
costs, fixed-bps slippage, and volume-aware diagnostics separately inspectable.

## 9. Required Future Report Fields

Future Markdown reports should include:

- title, date, experiment ID, and data scope.
- explicit synthetic/local-fixture caveat.
- input artifact list.
- split-window table.
- parameter-grid table.
- all-case, all-split summary table.
- invalid/insufficient-case table.
- benchmark assumption section.
- transaction-cost assumption section.
- fixed-slippage assumption section.
- volume-aware slippage diagnostic or precomputed-impact section, if relevant.
- guardrail section.
- next-step section that does not imply real-data validation or profitability.

## 10. Recommended Next Stage

After this plan is reviewed and merged, the next PR-sized stage should be:

```text
Synthetic split-aware robustness implementation
```

That stage should remain synthetic-only unless explicitly scoped otherwise. It
should add deterministic tests before or with implementation, report all
configured cases, avoid generated-output changes unless the PR explicitly
scopes them, and stop if it requires real data, vendor APIs, credentials,
live/paper trading, brokerage/order execution, or profitability framing.
